# 知识库路由规则说明

## 1. 目标

本说明用于统一知识平台在多知识库场景下的路由规则，避免检索、初始化、预处理和页面切换时出现“各接口各选各的库”的情况。

## 2. 路由优先级

知识库路由优先级如下：

1. 显式指定的 `knowledge_base_id`
2. 显式指定的 `knowledge_base_ids`
3. 当前激活知识库 `active_knowledge_base_id`

## 3. 检索路由规则

### 3.1 首选字段

检索请求优先使用 `knowledge_base_id`。

### 3.2 兼容字段

`knowledge_base_ids` 保留兼容，适合旧版本调用方式。

### 3.3 冲突规则

如果同时传入 `knowledge_base_id` 和 `knowledge_base_ids`，则：

- `knowledge_base_id` 必须包含在 `knowledge_base_ids` 中。
- 如果不包含，接口应返回 `400`，避免调用方传入互相矛盾的路由条件。

### 3.4 空值规则

如果两个字段都不传，则默认路由到当前激活知识库。

## 4. 初始化路由规则

初始化知识库时，必须显式指定目标 `knowledge_base_id`。

初始化动作包括：

- 创建知识库工作目录
- 创建默认 `pipeline_config.json`
- 初始化目录结构
- 让后续原始文件、chunk、向量、操作日志都能按库隔离

## 5. 页面切换规则

管理页面上切换知识库时，所有后续动作都应以当前选中的知识库为准，包括：

- 原始文件列表
- 原始文件上传
- 流程配置
- 预处理触发
- Chunk 生成
- Embedding 生成
- 检索测试

## 6. 开发约束

- 新接口优先支持单个 `knowledge_base_id`。
- 老接口中的 `knowledge_base_ids` 必须保留兼容。
- 任何“默认库”都只能作为兜底，不应成为业务调用主方式。
- 路由规则尽量集中在统一 helper 中实现，避免分散在各个接口里。

## 8. 调用示例

### 8.1 新版单库优先

```json
{
  "knowledge_base_id": "ai_qna_standard_v1",
  "query": "安全生产责任制的主要要求是什么",
  "top_k": 5,
  "limit": 5
}
```

### 8.2 兼容旧版列表传参

```json
{
  "knowledge_base_ids": ["ai_qna_standard_v1"],
  "query": "安全生产责任制的主要要求是什么",
  "top_k": 5,
  "limit": 5
}
```

### 8.3 冲突示例

```json
{
  "knowledge_base_id": "ai_qna_standard_v1",
  "knowledge_base_ids": ["ai_qna_standard_v2"],
  "query": "安全生产责任制的主要要求是什么"
}
```

这类请求会返回 `400`，因为单库 ID 不在允许列表中。
