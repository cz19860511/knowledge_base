# 多知识库管理方案

## 1. 目标

当前知识平台已经具备单知识库的原始文件管理、流程配置、预处理、Chunk、Embedding 和检索能力。下一步要升级为“多知识库管理平台”，让不同业务线、不同场景、不同 Agent 可以各自维护独立知识库，同时共用同一套平台能力。

多知识库模式的核心不是把多个库混在一起，而是做到：

- 一个知识库一套配置
- 一个知识库一套原始资料
- 一个知识库一套预处理和索引结果
- 一个知识库一组对外检索接口
- 一个知识库独立发布和回滚

## 2. 设计原则

### 2.1 数据隔离

每个知识库的原始文件、解析产物、chunk、向量库、发布包、运行日志都要隔离存放，避免互相覆盖。

### 2.2 配置隔离

每个知识库都要有自己的流程配置，包括：

- 预处理目录范围
- Chunk 参数
- Embedding 参数
- 检索参数

### 2.3 路由隔离

所有处理任务和检索请求都应该明确携带 `knowledge_base_id`，平台按 ID 路由到对应知识库。

### 2.4 页面隔离

用户进入平台后，先选知识库，再进入原始文件管理、流程配置、检索测试和发布管理页面。

### 2.5 发布隔离

每个知识库拥有自己的发布版本，不同知识库之间不能共享索引状态。

## 3. 统一抽象

建议把一个知识库抽象成一组稳定对象：

| 对象 | 说明 |
|---|---|
| `knowledge_base_id` | 知识库唯一标识 |
| `name` | 知识库名称 |
| `description` | 说明 |
| `owner` | 责任人 |
| `status` | active / paused / archived |
| `raw_root` | 原始文件目录 |
| `operations_root` | 运行台账和配置目录 |
| `selected_root` | 主文档目录 |
| `chunks_root` | chunk 输出目录 |
| `vectors_root` | 向量库目录 |
| `rag_root` | 检索测试和发布记录目录 |
| `pipeline_config` | 该库自己的流程配置 |

## 4. 推荐目录结构

建议从“单批次目录”升级为“多知识库目录 + 批次目录”。

```text
/data/kb/
  knowledge_bases/
    ai_qna_standard_v1/
      raw/
      raw_versions/
      operations/
      selected/
      chunks/
      vectors/
      rag/
      packages/
      metadata/
    retail_qna_v1/
      raw/
      raw_versions/
      operations/
      selected/
      chunks/
      vectors/
      rag/
      packages/
      metadata/
```

说明：

- `raw/` 存这个知识库的原始资料
- `raw_versions/` 存原始文件历史版本
- `operations/` 存流程配置、流水线状态、日志和台账
- `selected/` 存标准化后的主文档
- `chunks/` 存切片结果
- `vectors/` 存索引结果
- `rag/` 存检索测试、验证结果、发布摘要

## 5. 配置管理方式

### 5.1 全局注册表

建议增加一份平台级注册表，例如：

```text
/data/kb/operations/knowledge_bases.json
```

它负责记录所有已注册知识库及其基础信息。

示例字段：

- `knowledge_base_id`
- `name`
- `description`
- `status`
- `owner`
- `created_at`
- `updated_at`
- `default_batch_id`
- `root_dir`

### 5.2 库级配置

每个知识库保留自己的 `pipeline_config.json`。

建议路径：

```text
/data/kb/knowledge_bases/<kb_id>/operations/pipeline_config.json
```

如果暂时不改太多现有代码，也可以先继续用当前单库路径，后面再按 `kb_id` 升级。

## 6. 页面拆分

### 6.1 首页

首页不直接绑死一个知识库，而是做成“平台总览”。

首页应包含：

- 知识库列表
- 当前选中的知识库
- 各知识库状态
- 快捷入口

### 6.2 原始文件管理页

进入后先选知识库，再看该知识库的原始文件。

支持：

- 查看文件列表
- 上传
- 版本更新
- 删除
- 回滚
- 查看版本历史
- 手动触发预处理 / Chunk / Embedding

### 6.3 流程配置页

进入后先选知识库，再修改该知识库的流程参数。

支持：

- 预处理配置
- Chunk 配置
- Embedding 配置
- 保存后生效

### 6.4 检索测试页

进入后先选知识库，再发起检索测试。

支持：

- 按知识库切换
- 查看命中来源
- 查看规则命中情况
- 做回归测试

### 6.5 知识库总览页

建议增加一个“知识库管理”总览页，作为平台级入口。

展示内容：

- 知识库名称
- 知识库 ID
- 文档数
- chunk 数
- 索引版本
- 最近更新时间
- 当前状态

## 7. 接口拆分

### 7.1 平台级接口

- `GET /knowledge-bases`
- `POST /knowledge-bases`
- `PUT /knowledge-bases/{kb_id}`
- `DELETE /knowledge-bases/{kb_id}`
- `GET /knowledge-bases/{kb_id}/summary`

### 7.2 库级接口

- `GET /knowledge-bases/{kb_id}/raw-files`
- `POST /knowledge-bases/{kb_id}/raw-files/upload`
- `DELETE /knowledge-bases/{kb_id}/raw-files`
- `POST /knowledge-bases/{kb_id}/raw-files/rollback`
- `GET /knowledge-bases/{kb_id}/pipeline-config`
- `PUT /knowledge-bases/{kb_id}/pipeline-config`
- `POST /knowledge-bases/{kb_id}/pipeline`
- `POST /knowledge-bases/{kb_id}/retrieve`

### 7.3 兼容策略

为了不打断现有 AgentArts 联调，建议保留当前单库接口作为默认路由，同时逐步增加带 `kb_id` 的新接口。

## 8. 数据模型建议

### 8.1 KnowledgeBase

| 字段 | 说明 |
|---|---|
| `knowledge_base_id` | 知识库 ID |
| `name` | 名称 |
| `description` | 说明 |
| `owner` | 负责人 |
| `status` | 状态 |
| `root_dir` | 根目录 |
| `default_batch_id` | 默认批次 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 8.2 KnowledgeBaseVersion

| 字段 | 说明 |
|---|---|
| `version_id` | 版本 ID |
| `knowledge_base_id` | 所属知识库 |
| `batch_id` | 构建批次 |
| `pipeline_config` | 配置快照 |
| `index_manifest` | 索引摘要 |
| `status` | released / testing / rollback |

## 9. 实现顺序建议

### Phase 1：注册中心

先做知识库注册列表和基础总览。

### Phase 2：目录隔离

把原始文件、配置和运行结果从单库路径迁移到 `kb_id` 目录下。

### Phase 3：接口按库路由

让上传、配置、预处理、chunk、embedding、检索都支持 `kb_id`。

### Phase 4：页面切库

首页、原始文件页、流程配置页、检索页都支持选择知识库。

### Phase 5：发布管理

补齐发布、回滚、版本冻结和审计。

## 10. 当前项目建议

如果考虑交付节奏，建议先做最小可用版本：

1. 先实现知识库注册表
2. 再把现有 `ai_qna_standard_v1` 作为默认库保留
3. 再扩展第二个知识库
4. 最后做页面切换和目录隔离

这样可以保证现有 AgentArts 联调不被打断，同时逐步升级成真正的平台化能力。
