# 流程配置说明文档

本文说明知识平台 WebUI 中“流程配置”页面可配置的内容，包括 MinerU 预处理、Chunk 切分、Embedding 三类参数，以及这些参数对后续处理链路的实际影响。

## 1. 配置文件位置

流程配置页面读写同一份共享配置文件：

```text
/data/kb/operations/pipeline_config.json
```

本地开发时对应：

```text
/Users/chenzhuo/hb/knowledge_base/operations/pipeline_config.json
```

这份配置会被以下组件读取：

- `kb-api` 的流程配置接口 `GET /pipeline-config`、`PUT /pipeline-config`
- `preprocess_raw_02_07.py`
- `build_selected_and_chunks.py`
- `embedding_service`

## 2. 这份配置的作用

这份配置的目标不是替代全部环境变量，而是把**最常需要在页面上调整、并且会直接影响处理结果**的参数统一管理起来。

它主要解决四件事：

- 决定本轮哪些原始目录参与处理
- 决定预处理阶段使用哪些解析方式
- 决定 chunk 怎么切
- 决定 embedding 服务怎么跑

## 3. MinerU 预处理配置

### 3.1 `folders`

作用：

- 指定本轮要处理的原始资料目录
- 预处理脚本、Chunk 脚本都会按这个目录集合工作
- 页面上勾选哪些目录，后面执行链路就只针对这些目录

建议：

- 默认保持 `02-07` 全选
- 如果只想重跑某一类资料，可以只勾选单个目录

影响范围：

- `preprocess_raw_02_07.py`
- `build_selected_and_chunks.py`

### 3.2 `primary_exts`

作用：

- 定义 MinerU 优先处理的文件扩展名
- 当前默认是：`.pdf`、`.docx`、`.xlsx`、`.pptx`

为什么要可配：

- 有些资料类型未来可能需要加入或移除
- 不同批次可能会出现新的主格式

影响范围：

- 预处理脚本在扫描源文件时使用

### 3.3 `supplement_exts`

作用：

- 定义要额外做补充解析的文件类型
- 当前默认只对 `.docx` 做 `MarkItDown` 补充

为什么要补充解析：

- MinerU 是主解析器
- 但对某些 `docx` 文件，`MarkItDown` 会给出更平滑的文本版本
- 后续选稿时可以在 MinerU 和补充解析之间做对比

影响范围：

- `preprocess_raw_02_07.py`

### 3.4 `markitdown_docx_enabled`

作用：

- 控制是否启用 DOCX 的 `MarkItDown` 补充解析

适用场景：

- 如果希望只跑 MinerU，可关闭
- 如果想保留 `docx` 的备份文本稿，建议开启

影响范围：

- 预处理脚本执行时是否生成 `working/parsed/.../markitdown_docx/`

### 3.5 `mineru_command`

作用：

- 配置调用 MinerU 的命令名
- 默认是 `mineru`

适用场景：

- 本地和 ECS 环境中 MinerU 安装路径不一致时
- 如果后续切换成包装脚本或别名命令，也可以在这里调整

影响范围：

- 预处理脚本执行外部命令时使用

### 3.6 `mineru_pipeline`

作用：

- 配置 MinerU 的 `-b` 参数
- 当前默认是 `pipeline`

适用场景：

- MinerU 的执行模式名称发生变化时
- 不同版本 MinerU 需要切换管线名时

影响范围：

- 预处理脚本构造 MinerU 命令行时使用

## 4. Chunk 切分配置

### 4.1 `folders`

作用：

- 指定本次 chunk 重建要处理哪些目录
- 一般与预处理目录保持一致

为什么要单独保留：

- 有时会出现“只重切某个目录”的场景
- 这样可以避免全量重算

影响范围：

- `build_selected_and_chunks.py`

### 4.2 `max_chunk_chars`

作用：

- 控制单个 chunk 的最大字符数
- 当前默认值：`1800`

影响：

- 值越大，chunk 越长，上下文更完整
- 值越小，chunk 更细，命中更精确但容易丢上下文

建议：

- 适合中文知识问答的常见范围通常在 `1200 ~ 2500`
- 如果文档结构清晰、章节很短，可以适当调小
- 如果文档内容连贯、条款较长，可以适当调大

### 4.3 `min_chunk_chars`

作用：

- 控制切分时“最小保留长度”
- 当前默认值：`300`

影响：

- 避免 chunk 被切得过碎
- 保持一定的语义完整性

建议：

- 不宜过低
- 太低会导致大量碎片化 chunk，影响检索效果

### 4.4 `mineru_parser_bonus`

作用：

- 当同一份文档存在多个 Markdown 候选时，用来提高 MinerU 结果的优先级
- 当前默认值：`50`

为什么有这个参数：

- 我们的主解析器是 MinerU
- 但某些文档也会有 MarkItDown 或其他候选结果
- 这个分值让系统在质量接近时优先选 MinerU

适用场景：

- 想更偏向 MinerU 的结构化输出时
- 想让补充解析更多只作为兜底时

## 5. Embedding 配置

### 5.1 `provider`

作用：

- 指定 embedding 服务使用的实现
- 当前支持：
  - `transformers`
  - `mock`

说明：

- `transformers` 是正式推理模式
- `mock` 是服务联调和空模型测试用

影响范围：

- `embedding_service`

### 5.2 `model_path`

作用：

- 指定本地 embedding 模型目录
- 当前默认值：`/data/kb/models/bge-small-zh-v1.5`

影响：

- 决定服务加载哪一个中文向量模型
- 只要目录中模型文件齐全，服务就会按这个路径加载

### 5.3 `model_name`

作用：

- 展示当前模型名称
- 会回显到 `/health`、`/models` 和检索返回中

说明：

- 它更多是“标识字段”
- 真正加载模型仍然看 `model_path`

### 5.4 `device`

作用：

- 指定模型运行设备
- 常见值：
  - `cpu`
  - `cuda`

建议：

- 当前 ECS 版本默认使用 `cpu`
- 如果后续部署到 GPU 环境，可以切到 `cuda`

### 5.5 `batch_size`

作用：

- 控制 embedding 推理时每批处理多少条文本

影响：

- 值大：吞吐更高，但更吃内存
- 值小：更稳，但速度慢一些

建议：

- 服务器内存比较稳时可以适当调大
- 如果遇到推理卡顿或内存压力，可以先调小

### 5.6 `pooling`

作用：

- 决定把 token 级输出聚合成句向量的方式

当前支持：

- `cls`
- `mean`

说明：

- `cls` 取首 token 表示，通常速度快、实现简单
- `mean` 是对 token 结果做平均，更平滑，但有时更依赖输入文本质量

当前推荐：

- 先保持 `cls`

### 5.7 `query_instruction`

作用：

- 当输入类型是 `query` 时，自动给查询文本加前缀提示
- 这对中文检索向量通常很重要

为什么需要：

- embedding 模型通常对“文档”和“查询”有不同的表达习惯
- 加 instruction 有助于让查询向量更贴近检索任务

当前默认：

- `为这个句子生成表示以用于检索相关文章：`

### 5.8 `max_length`

作用：

- 控制 embedding 输入截断长度

影响：

- 值越大，能保留更多文本
- 值越大，推理也更慢，显存/内存压力更高

建议：

- 中文知识问答场景下，`512` 是比较稳妥的默认值

### 5.9 `normalize`

作用：

- 控制 embedding 输出是否做 L2 归一化

影响：

- 开启后，向量更适合做余弦相似度类检索
- 当前知识平台默认开启

建议：

- 保持 `true`

## 6. 修改后什么时候生效

### 6.1 预处理配置

生效方式：

- 保存配置后，下一次执行预处理脚本时生效

说明：

- 不需要重启 WebUI
- 但需要重新触发预处理链路

### 6.2 Chunk 配置

生效方式：

- 保存配置后，下一次执行 chunk 脚本时生效

说明：

- 需要重新跑 `Chunk` 阶段，新的切分规则才会应用到结果里

### 6.3 Embedding 配置

生效方式：

- 保存配置后，embedding 服务会在下一次请求时自动读取

说明：

- 目前 embedding 服务会按配置文件自动刷新
- 一般不需要手工重启服务

## 7. 推荐的调整顺序

如果要调优，建议按这个顺序来：

1. 先调 `folders`
2. 再调 `max_chunk_chars` 和 `min_chunk_chars`
3. 再调 `query_instruction`
4. 最后再考虑 `batch_size`、`pooling` 和 `device`

原因：

- 目录和 chunk 规则会直接改变检索单元
- embedding 参数更多是推理性能和向量表达方式的调整

## 8. 当前推荐默认值

当前建议先保持这组默认值：

- `preprocess.folders`：`02-07` 全选
- `preprocess.primary_exts`：`.pdf, .docx, .xlsx, .pptx`
- `preprocess.supplement_exts`：`.docx`
- `preprocess.markitdown_docx_enabled`：`true`
- `preprocess.mineru_command`：`mineru`
- `preprocess.mineru_pipeline`：`pipeline`
- `chunk.max_chunk_chars`：`1800`
- `chunk.min_chunk_chars`：`300`
- `chunk.mineru_parser_bonus`：`50`
- `embedding.provider`：`transformers`
- `embedding.model_path`：`/data/kb/models/bge-small-zh-v1.5`
- `embedding.model_name`：`bge-small-zh-v1.5`
- `embedding.device`：`cpu`
- `embedding.batch_size`：`16`
- `embedding.pooling`：`cls`
- `embedding.query_instruction`：保持默认中文检索前缀
- `embedding.max_length`：`512`
- `embedding.normalize`：`true`

## 9. 相关页面和接口

- 流程配置页面：`/pipeline-config-ui`
- 配置读取接口：`GET /pipeline-config`
- 配置保存接口：`PUT /pipeline-config`
- 首页入口：`/`
- 原始文件管理页：`/raw-files-ui`

## 10. 小结

这份配置的核心价值是把“知识平台最常调、最影响结果”的参数收口到一个地方，方便现场快速调整，也方便后续版本管理。

如果后面要继续平台化，建议继续沿着这份配置扩展，但保持原则不变：

- 页面可见
- 文件可追踪
- 修改可回放
- 生效边界清楚
