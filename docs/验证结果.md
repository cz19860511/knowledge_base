# 中文 Embedding 模型小型选型验证结果

版本：v0.3  
日期：2026-05-22  
范围：AI+智能问答智能体知识平台  
当前状态：已完成 `BAAI/bge-small-zh-v1.5` 与 `BAAI/bge-base-zh-v1.5` 本地首轮验证；当前推荐路线已收敛为 `bge-small + 检索策略增强`，本地索引已切回 small，ECS 生产服务尚未同步部署。

## 1. 验证结论

本轮验证属于“小型模型选型验证”，重点是用同一批业务问题对比当前基线、`bge-small-zh-v1.5` 和 `bge-base-zh-v1.5`。

当前结论如下：

| 事项 | 结论 |
|---|---|
| 是否已最终选定中文 embedding 模型 | 首版选定 `BAAI/bge-small-zh-v1.5` |
| 当前本地可用检索方案 | `关键词检索 + bge-small embedding + 业务规则增强` 的 hybrid 检索 |
| 是否已完成真实中文 embedding 实测 | 已完成 small/base 本地实测 |
| 首版默认模型 | `BAAI/bge-small-zh-v1.5` |
| 对照验证模型 | `BAAI/bge-base-zh-v1.5`，效果没有显著优于 small，暂不作为首版默认 |
| 当前增强方向 | 查询扩展、关键词强召回、场景规则加权、业务资料补齐 |
| 后续增强候选 | `BAAI/bge-m3` |
| 暂不建议首版直接上线 | `bge-m3`，功能强但模型更重，首版上线成本和延迟风险更高 |

建议首版正式走“小模型可控上线 + 检索策略增强”路线：采用 `bge-small-zh-v1.5` 作为默认 embedding 模型，同时保留关键词检索，并增加场景规则增强。`bge-base-zh-v1.5` 体积和延迟更高，但在本轮关键问题上没有解决 small 的短板，暂不建议作为首版默认模型。

## 3. 候选模型对比

| 模型 | 定位 | 预期优势 | 主要风险 | 首轮建议 |
|---|---|---|---|---|
| `BAAI/bge-small-zh-v1.5` | 中文轻量 embedding | 中文制度类问答足够稳，CPU 成本相对可控，适合首版上线 | 复杂语义和长文本召回可能弱于 base/m3 | 首选 |
| `BAAI/bge-base-zh-v1.5` | 中文质量增强 embedding | 召回质量通常优于 small，适合做第二候选 | CPU 延迟、内存和建库耗时更高 | 对照验证 |
| `BAAI/bge-m3` | 多语言、多粒度、多功能 embedding | 支持更长文本和更复杂检索形态，后续平台化空间大 | 模型更重，首版部署和延迟压力更大 | 后续增强 |
| `moka-ai/m3e-base` | 中文通用 embedding | 中文生态里使用较多，可作为历史对照 | 相比 BGE v1.5/M3，后续维护和效果需实测确认 | 备选对照 |

说明：模型维度、最大长度、pooling 方式以后续下载后的 `config.json` 和模型卡为准。尤其是 BGE 类模型，验证时必须按模型卡推荐方式做 pooling 和 normalize，否则检索结果不可作为正式结论。

## 4. 验证目标

本轮验证不是跑大规模榜单，而是回答三个项目问题：

| 问题 | 验证方式 |
|---|---|
| 哪个模型适合第一版上线 | 看 Top1/Top3 命中、延迟、内存、建库耗时 |
| 当前 hybrid 是否需要保留关键词检索 | 对比纯 embedding 与 hybrid 的命中稳定性 |
| 首版是否需要 rerank | 如果 Top3 召回可以但 Top1 排序差，再考虑 rerank |

## 8. BGE Small/Base 首轮实测结果

### 8.1 模型与索引

| 模型 | 本地目录大小 | 权重文件 | Embedding 维度 | 建库状态 |

| `BAAI/bge-small-zh-v1.5` | 约 `92MB` | `model.safetensors` | `512` | 已完成 |
| `BAAI/bge-base-zh-v1.5` | 约 `400MB` | `pytorch_model.bin` | `768` | 已完成 |

两轮均使用相同设置：

- 文档 embedding：`input_type=document`，不加 query instruction。
- 查询 embedding：`input_type=query`，使用 `为这个句子生成表示以用于检索相关文章：` 作为 query instruction。

- Normalize：开启。
- Hybrid 权重：关键词 `0.60`，embedding `0.40`；命中场景规则时叠加规则权重 `0.20`，最终按有效权重归一。

| 模型 | 首次查询 | 预热后单次查询 | 判断 |

### 8.4 关键发现

| 发现 | 说明 |
|---|---|
| bigger model 未显著解决核心问题 | `bge-base` 对 Q02/Q04 没有明显优于 `bge-small` |
| Q02/Q04 更像检索策略问题 | 知识库中有危化品和收银资料，但完整业务问法会被泛化词带偏 |
| 关键词强召回仍必须保留 | 直接检索“危化品”可以命中正确资料，说明关键词和实体召回要加强 |
| 需要补业务规则资料 | “违规收银如何判断”需要明确规则、判断条件、处置流程，现有资料偏岗位职责或通用制度 |
| 首版不宜直接上 base | base 体积约 4 倍，延迟更高，但本轮质量收益不足 |

## 8.5 Small + 检索策略增强复测结果

本轮已将首版路线调整为 `bge-small + 检索策略增强`，增强内容包括：

- 查询扩展：对业务问法补充常见实体、制度关键词和现场处置词。
- 强规则召回：对危化品车辆、拥堵疏导、违规收银、消防检查、设备设施巡检设置场景规则。
- 规则加权：命中场景规则时，在关键词分数和 embedding 分数之外叠加 `rule_score`。
- 安全降级：embedding 服务不可用时，`kb-api` 可退回 `keyword + rules`，不影响基础检索可用性。

本地复测环境：

| 项目 | 值 |
|---|---|
| Embedding 模型 | `BAAI/bge-small-zh-v1.5` |
| 文档 embedding 维度 | `512` |
| 当前索引规模 | `3053` chunks / `199` docs |
| 检索模式 | `keyword + embedding + rules` |
| 默认权重 | keyword `0.60` / embedding `0.40` / rules `0.20` |

## 10. 技术注意事项

| 注意事项 | 影响 |
|---|---|
| pooling 方式必须与模型卡一致 | pooling 不一致会导致 embedding 效果失真 |
| 文档 embedding 和 query embedding 必须使用同一模型 | 否则向量空间不一致，结果不可用 |
| 维度必须一致 | 例如文档向量 512 维，query 向量也必须 512 维 |
| 仍需保留关键词检索 | 制度编号、岗位名称、专有名词依赖关键词召回 |
| 不建议首版直接上 rerank | 先看 Top3 召回，召回足够但排序差时再引入 rerank |
| 模型不要进 Git | 模型放 `/data/kb/models/` 或对象存储 |

## 11. 本轮推荐

建议按以下顺序推进：

1. 首版模型选定 `BAAI/bge-small-zh-v1.5`。
2. 检索方案采用 `关键词检索 + bge-small embedding + 场景规则增强`。
3. 暂不把 `bge-base-zh-v1.5` 作为默认生产模型。
4. 对 Q03、Q04、Q07、Q10 做第二轮规则排序调优。
5. 如果后续文档扩展到多语言、超长文本、多粒度检索，再评估 `BAAI/bge-m3`。

阶段性结论：首版上线模型采用 `BAAI/bge-small-zh-v1.5`，通过检索策略增强弥补小模型在现场业务问法上的短板。`bge-base-zh-v1.5` 可作为备用对照，但当前不值得为其增加首版部署成本。
