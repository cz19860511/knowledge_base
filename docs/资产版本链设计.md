# 资产版本链设计

## 1. 目标

把知识平台中的所有关键产物统一纳入版本链，做到“每个文件都知道从哪来、被谁改过、当前是哪一版、能不能回滚”。

## 2. 资产范围

当前先覆盖以下几类资产：

- 原始文件
- 预处理输出
- selected 主版本文档
- chunk 产物
- 向量索引和向量文件
- 平台运行日报
- 平台运行记忆库产物
- 自进化建议报告

## 3. 统一字段

建议所有资产记录都带以下字段：

- `asset_id`
- `knowledge_base_id`
- `asset_type`
- `stage`
- `logical_path`
- `version`
- `status`
- `file_path`
- `checksum`
- `size_bytes`
- `created_by`
- `created_at`
- `operation_id`
- `parent_asset_id`
- `source_asset_ids`
- `metadata`

## 4. 版本规则

- 同一 `knowledge_base_id + asset_type + stage + logical_path` 视为同一个资产链。
- 每次产生新文件或新结果，就追加一条新版本记录。
- 如果是回滚、删除或重跑，也要写一条新记录，不覆盖旧记录。
- `parent_asset_id` 用来串起版本链，方便追溯前后关系。

## 5. 当前实现

当前已经落地的部分包括：

- 原始文件上传、回滚、删除会写入资产版本链。
- 预处理 `working/parsed/` 产物会写入资产版本链。
- `selected/` 主版本文档、候选清单会写入资产版本链。
- `chunks/` 切片产物和预览文件会写入资产版本链。
- `vectors/` 向量、索引、manifest 和包文件会写入资产版本链。
- 平台运行日报、selected、chunk、vector 产物会写入资产版本链。
- 可通过 `/operations/assets` 查询资产版本列表。

## 6. 设计原则

- 资产台账是追加式，不做原地覆盖。
- 版本链和操作事件要互相对应。
- 中间产物和最终产物都要保留。
- 所有版本记录都要尽量能回溯到原始来源。
