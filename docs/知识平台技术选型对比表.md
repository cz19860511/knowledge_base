# 知识平台技术选型对比表

## 1. 选型结论

建议采用“稳态底座 + 可替换插件”的路线。

第一阶段以 `FastAPI + PostgreSQL + Redis + Celery + OSS/MinIO + Docker Compose` 作为平台基础底座。检索层先保留当前可用的 TF-IDF 方案作为基线，同时预留向量库、关键词检索和 rerank 的插件接口。

后续随着数据规模和检索质量要求提升，再逐步引入 `pgvector / Milvus / OpenSearch / Elasticsearch / rerank model` 等能力。

## 2. 总体推荐组合

| 层级 | 第一阶段推荐 | 后续增强方向 | 选择理由 |
|---|---|---|---|
| Web/API 服务 | FastAPI | FastAPI 继续保留 | 与当前 kb-api 一致，OpenAPI 友好，开发快 |
| 元数据存储 | PostgreSQL | PostgreSQL + 分区/读写分离 | 适合文档、版本、任务、发布、审计等关系数据 |
| 文件存储 | OSS 或 MinIO | OSS/MinIO 多环境统一 | 适合原始文件、解析产物、发布包和归档 |
| 任务队列 | Celery + Redis | Temporal / Airflow | 第一阶段足够快，复杂编排再升级 |
| 预处理 | MinerU + Marker + MarkItDown | 增加解析器插件 | 与当前方案一致，保留可替换空间 |
| Chunk | 自研结构化切块服务 | 规则 + 模型辅助切块 | 业务资料需要按章节、条款、表格切 |
| Embedding | 本地或云端 embedding 服务 | 多模型路由 | 先接入一种稳定模型，后续支持切换 |
| 向量库 | pgvector 或 FAISS | Milvus | 小规模优先简单，规模上来再上专用向量库 |
| 关键词检索 | PostgreSQL FTS / BM25 | OpenSearch / Elasticsearch | 制度资料需要精确词和编号召回 |
| 混合检索 | 自研 merge/rank | 专用检索服务 | 保持策略可控，后续可服务化 |
| Rerank | 暂不上或轻量模型 | BGE reranker / 云端 rerank | 第一阶段先验证收益，再引入成本 |
| Adapter | AgentArts General | 多平台 adapter | AgentArts 是第一个消费者，不写死核心模型 |
| 部署 | Docker Compose | Kubernetes | 当前团队落地快，后续再规模化 |
| 监控 | 应用日志 + 基础指标 | Prometheus + Grafana | 第一阶段先可观察，后续再平台化 |

## 3. API 服务选型

| 方案 | 优点 | 风险/限制 | 适用阶段 | 建议 |
|---|---|---|---|---|
| FastAPI | Python 生态友好，OpenAPI 自动生成，和当前 kb-api 一致 | 高并发下需要配合 worker 和网关 | MVP 到生产 | 推荐 |
| Flask | 简单轻量 | OpenAPI、类型校验和异步支持不如 FastAPI 顺手 | 小工具 | 不作为主选 |
| Spring Boot | 企业级生态成熟，治理能力强 | 当前知识处理链路以 Python 为主，开发成本更高 | 大型 Java 团队 | 暂不优先 |
| Node.js/NestJS | 工程化强，前后端团队容易接 | 文档解析和 ML 生态不如 Python 直接 | 前端团队主导时 | 可作为前端管理台后端 |

推荐：继续沿用 `FastAPI`，把当前 `kb-api` 升级成统一服务层。

## 4. 元数据存储选型

| 方案 | 优点 | 风险/限制 | 适用阶段 | 建议 |
|---|---|---|---|---|
| PostgreSQL | 关系模型强，事务稳定，支持 JSONB 和全文检索 | 运维需要基础 DBA 能力 | MVP 到生产 | 推荐 |
| MySQL | 团队熟悉，部署简单 | JSON、全文检索和后续 pgvector 扩展不如 PostgreSQL | 只做基础台账 | 可用但不优先 |
| SQLite | 简单，无服务依赖 | 并发、权限、远程管理不足 | 原型验证 | 只用于本地原型 |
| MongoDB | 文档模型灵活 | 版本、任务、发布审计这类关系数据不如 PostgreSQL 清晰 | 非结构化元数据较多 | 暂不优先 |

推荐：平台正式版本采用 `PostgreSQL`。当前 SQLite 索引可以作为过渡，不作为长期主存储。

## 5. 文件与对象存储选型

| 方案 | 优点 | 风险/限制 | 适用阶段 | 建议 |
|---|---|---|---|---|
| 阿里云 OSS | 云上成熟，适合 ECS 部署，权限和生命周期管理完善 | 依赖云资源和账号配置 | 生产 | 推荐 |
| MinIO | 私有化友好，S3 兼容，便于本地/内网部署 | 需要自行运维 | 私有化和测试 | 推荐作为兼容方案 |
| 本地磁盘 | 快速简单 | 扩展、备份、权限和迁移风险较高 | 原型 | 仅用于过渡 |
| NAS | 共享访问方便 | 成本和运维取决于云环境 | 大文件共享 | 可选 |

推荐：抽象为统一对象存储接口。云上优先 `OSS`，私有化环境优先 `MinIO`。

## 6. 任务编排选型

| 方案 | 优点 | 风险/限制 | 适用阶段 | 建议 |
|---|---|---|---|---|
| Celery + Redis | Python 生态成熟，适合异步任务和批处理 | 复杂工作流状态管理一般 | MVP 到生产初期 | 推荐 |
| RQ + Redis | 简单轻量 | 编排能力弱 | 小规模任务 | 可用于原型 |
| Airflow | DAG 编排强，适合周期任务 | 交互式任务和实时反馈较重 | 数据工程化阶段 | 后续可选 |
| Temporal | 工作流可靠性强，支持长流程状态 | 学习和接入成本较高 | 复杂平台阶段 | 后续增强 |
| Argo Workflows | K8s 原生，适合容器任务 | 依赖 K8s | 云原生阶段 | 后续可选 |

推荐：第一阶段用 `Celery + Redis`，等流程复杂到需要强状态编排时再评估 Temporal。

## 7. 预处理与解析选型

| 工具 | 定位 | 优点 | 风险/限制 | 建议 |
|---|---|---|---|---|
| MinerU | 主解析器 | 适合 PDF、扫描件、复杂版式 | 依赖较重，运行耗时高 | 主线使用 |
| Marker | 抽样对比和兜底 | 多格式支持，适合质量对比 | 与 MinerU 结果可能差异大 | 抽样和失败兜底 |
| MarkItDown | DOCX 补充 | Office 文档转换快 | 复杂版式和表格能力有限 | DOCX 快速补充 |
| python-docx/openpyxl | 结构化读取 | 对 Word/Excel 字段抽取可控 | 需要自定义规则 | 表单和字段类资料使用 |

推荐：保持当前策略，即 `MinerU 主线，Marker 抽样对比，MarkItDown 补 DOCX`。

## 8. Chunk 策略选型

| 方案 | 优点 | 风险/限制 | 适用资料 | 建议 |
|---|---|---|---|---|
| 固定长度切块 | 实现简单，速度快 | 容易切断条款和上下文 | 原型 | 仅作为兜底 |
| 标题/章节切块 | 可解释性强，引用清晰 | 依赖解析结构质量 | 制度、SOP、手册 | 推荐 |
| 条款级切块 | 精准，适合问答引用 | 条款识别需要规则 | 标准、制度、规范 | 推荐 |
| 表格结构切块 | 保留字段和表头 | 需要表格抽取能力 | 表单、台账、字段说明 | 推荐 |
| 语义切块 | 上下文自然 | 需要模型或复杂规则 | 长段落材料 | 后续增强 |

推荐：采用混合 chunk 策略。制度和 SOP 按章节/条款切，表单按表格结构切，长文本再做长度保护。

## 9. Embedding 模型选型

| 方案 | 优点 | 风险/限制 | 适用阶段 | 建议 |
|---|---|---|---|---|
| 本地中文 embedding 模型 | 数据不出域，成本可控 | 需要 GPU 或较好 CPU 资源 | 私有化和稳定运行 | 推荐评估 |
| 云端 embedding API | 接入快，效果稳定 | 成本、合规和网络依赖 | 快速验证 | 可用于对照测试 |
| 多语言 embedding 模型 | 中英文兼容 | 中文制度资料可能不是最优 | 多语种资料 | 备用 |
| 当前 TF-IDF | 零模型依赖，解释简单 | 语义能力弱 | 原型和兜底 | 保留为基线 |

推荐：第一阶段先保留 TF-IDF 作为关键词基线，同时引入一个中文 embedding 模型做 A/B 对照。模型选择应通过真实问题集评估，而不是只看榜单。

## 10. 向量库选型

| 方案 | 优点 | 风险/限制 | 适用规模 | 建议 |
|---|---|---|---|---|
| FAISS | 本地高性能，部署简单 | 元数据、权限、分布式能力弱 | 原型到小规模 | 可作为过渡 |
| pgvector | 和 PostgreSQL 集成，元数据和向量同库管理 | 超大规模性能需评估 | 小到中等规模 | 推荐第一阶段重点评估 |
| Milvus | 专业向量库，扩展能力强 | 部署和运维成本较高 | 中大型规模 | 后续扩展推荐 |
| Chroma | 上手快，适合原型 | 生产治理能力需评估 | 原型和小规模 | 原型可用 |
| Elasticsearch/OpenSearch kNN | 可同时做搜索和向量 | 配置和成本较高 | 搜索场景强 | 后续可选 |

推荐：

| 阶段 | 选择 |
|---|---|
| 当前原型 | TF-IDF + SQLite 已可保留 |
| 平台 MVP | PostgreSQL + pgvector |
| 规模化 | Milvus 或 OpenSearch/Elasticsearch kNN |

## 11. 关键词检索选型

| 方案 | 优点 | 风险/限制 | 适用阶段 | 建议 |
|---|---|---|---|---|
| PostgreSQL FTS | 与元数据同库，简单 | 中文分词能力需要配置 | MVP | 可用 |
| BM25 自建 | 策略可控，成本低 | 工程维护成本 | MVP 到生产初期 | 推荐作为基线 |
| OpenSearch | 搜索能力强，生态成熟 | 运维和成本更高 | 中后期 | 推荐后续评估 |
| Elasticsearch | 能力成熟，插件丰富 | 商业和授权策略需确认 | 中后期 | 可选 |

推荐：第一阶段保留 BM25/TF-IDF 基线；如果查询量和文档规模上来，再引入 OpenSearch。

## 12. Rerank 选型

| 方案 | 优点 | 风险/限制 | 适用阶段 | 建议 |
|---|---|---|---|---|
| 不使用 rerank | 简单，延迟低 | 排序质量有限 | 原型 | 当前可接受 |
| 本地 rerank 模型 | 效果提升明显，数据不出域 | 需要算力，延迟增加 | 质量优化阶段 | 推荐评估 |
| 云端 rerank API | 接入快 | 成本和合规依赖 | 快速验证 | 可做对照 |
| 规则 rerank | 可解释，稳定 | 泛化能力有限 | 制度编号、版本优先级 | 推荐叠加 |

推荐：第二阶段引入 rerank。先对真实问题集测试收益，再决定是否进入生产。

## 13. 外部适配选型

| 消费方 | 接入方式 | 优点 | 注意事项 |
|---|---|---|---|
| AgentArts | General 第三方知识库 | 已完成接口原型，路径清晰 | 返回字段需严格符合官方规范 |
| 内部 Agent | 统一检索 API | 可按业务定制权限和返回格式 | 需要调用方身份识别 |
| 业务系统 | REST API | 接入通用 | 需要限流、鉴权和审计 |
| RAGFlow | RAGFlow 适配器 | 如果后续采用 RAGFlow，可对接 | 当前本地库不应强行迁移 |
| OpenClaw/其他平台 | Adapter | 不影响核心知识平台 | 需要统一 canonical schema |

推荐：核心检索 API 自主定义，所有外部平台通过 adapter 转换。

## 14. 权限与安全选型

| 能力 | 第一阶段 | 后续增强 |
|---|---|---|
| API 鉴权 | API Key | OAuth2 / JWT / AK-SK |
| 调用方隔离 | 每个平台独立 key | 租户、项目、Agent 维度隔离 |
| 知识权限 | 知识域级过滤 | 文档级、字段级、角色级过滤 |
| 日志审计 | 记录 query、调用方、命中结果 | 敏感词、脱敏、异常告警 |
| 网络安全 | Nginx + 安全组 | HTTPS、WAF、专线/VPN |

推荐：第一阶段使用 API Key，生产接入前至少启用 HTTPS 和调用日志。

## 15. 部署选型

| 方案 | 优点 | 风险/限制 | 适用阶段 | 建议 |
|---|---|---|---|---|
| Docker Compose | 简单直接，适合 ECS | 多实例和弹性能力有限 | MVP 和小规模 | 推荐 |
| Kubernetes | 扩展、发布和治理能力强 | 运维复杂度高 | 规模化平台 | 后续 |
| 裸机 Python 服务 | 快速 | 环境漂移严重 | 临时验证 | 不建议生产 |
| Serverless | 运维轻 | 长任务和大文件处理不合适 | 轻量 API | 暂不优先 |

推荐：当前继续 `Docker Compose`，把数据和服务解耦。后续需要多节点和弹性时迁移到 K8s。

## 16. 最小可行技术路线

```text
文件管理
  PostgreSQL 记录文件主数据
  OSS/MinIO 保存原始文件和解析产物

任务编排
  FastAPI 创建任务
  Celery + Redis 执行预处理、chunk、embedding、索引构建

知识加工
  MinerU / Marker / MarkItDown 解析
  结构化 chunk
  TF-IDF + embedding 双路索引

检索发布
  统一检索 API
  AgentArts General adapter
  发布版本和回滚记录
```

## 17. 当前到目标的迁移路径

| 当前能力 | 迁移动作 | 目标能力 |
|---|---|---|
| 本地 raw 目录 | 接入文件管理模块 | 文件可上传、可同步、可追踪 |
| 预处理脚本 | 封装为任务 worker | 可排队、可重试、可查看状态 |
| chunks.jsonl | 写入 chunk 表和对象存储 | chunk 可查询、可追溯、可重建 |
| SQLite/TF-IDF | 保留基线并接入 pgvector | 支持关键词和语义混合检索 |
| kb-api | 升级为检索服务和 adapter | 支持多平台、多 Agent 调用 |
| ECS Docker Compose | 保留为 MVP 部署 | 后续迁移到 K8s 或多节点 |

## 18. 近期建议优先级

| 优先级 | 工作 | 原因 |
|---|---|---|
| P0 | 建立 PostgreSQL 文档主数据表 | 没有主数据，后续版本和发布不可控 |
| P0 | 设计 canonical schema | 决定平台兼容性和 adapter 扩展性 |
| P0 | 把预处理脚本封装成任务 | 平台化的第一步 |
| P1 | 引入结构化 chunk 策略 | 直接影响检索质量 |
| P1 | 评估 pgvector 和 embedding 模型 | 为语义检索升级做准备 |
| P1 | 建立回归问题集 | 没有评估集，无法判断升级收益 |
| P2 | 增加 rerank | 在召回稳定后提升排序 |
| P2 | 增加发布回滚 | 进入生产前必须具备 |

## 19. 推荐决策

| 决策项 | 推荐结论 |
|---|---|
| 平台定义 | 知识平台建设 |
| 第一阶段底座 | FastAPI + PostgreSQL + Redis + Celery + OSS/MinIO |
| 当前检索 | 保留 TF-IDF 作为基线 |
| 下一步检索升级 | pgvector + 中文 embedding + BM25 |
| 后续规模化 | Milvus 或 OpenSearch/Elasticsearch |
| 对外接入 | 统一 API + adapter，AgentArts 是第一个 adapter |
| 部署方式 | Docker Compose 起步，未来 K8s |
