# AI+智能问答知识库搭建实施计划 v2

## 1. 重新梳理的结论

旧版方案的技术路线有价值，尤其是“文档预处理、解析质量评估、标准化 Markdown、切片、向量化入库、RAG 检索、AgentArts 调用”这条链路。但如果直接按旧方案推进，容易把知识库做成“大而全的文档处理平台”，而不是能支撑 `AI+智能问答智能体` 稳定交付的知识资产。

新版计划要先聚焦四件事：前期资料预处理、生成 chunk、生成向量库、RAG 检索。资料治理、元数据、质量评估和 AgentArts 接入都围绕这条主链路服务，暂时不铺开完整文档中台。

第一阶段目标不是做一个完整文档中台，而是交付一个可被 AgentArts 调用、能回答标准化制度和 SOP 问题、答案可追溯、资料可维护、RAG 检索效果可验证的知识库。

## 2. 第一阶段建设目标

第一阶段只服务 `AI+智能问答智能体`，优先覆盖服务区标准化运营相关知识。

核心目标：

1. 支持制度、标准、SOP、岗位职责、应急安全、APP 操作等资料入库。
2. 以前期资料预处理为起点，形成稳定的标准化 Markdown。
3. 按文档结构生成可追溯 chunk。
4. 生成第一版向量库。
5. 支持 RAG 检索，并返回来源文档、版本、章节或页码线索。
6. 能形成可回归测试的问题集和命中评估结果。
7. 能输出 AgentArts 可使用的知识包或接口清单。

第一阶段暂不做：

1. 复杂业务系统写回。
2. 自动任务派发和事件闭环。
3. 完整知识图谱。
4. 全自动无人审核入库。
5. 大规模多租户知识库平台。
6. 大规模 LLM 二次结构化抽取。

## 3. 新版总体流程

```text
raw 原始资料
  ↓
资料盘点与定版
  ↓
元数据补齐
  ↓

  ↓

  ↓
解析质量评估
  ↓

  ↓

  ↓

  ↓
RAG 检索测试
  ↓
AgentArts 知识包 / 知识库接口
  ↓
上线反馈与持续更新
```


目录原则：

1. `raw` 永远只放原始文件，不做编辑。
2. `inventory` 管资料盘点和版本。
3. `metadata` 管知识库统一口径。
4. `working` 放机器处理过程产物，可重跑。
5. `selected` 放人工确认后的可入库主版本。
6. `chunks` 放最终进入检索的知识切片。
7. `qa` 放测试问题和回归结果。
8. `packages` 放对 AgentArts 或其他平台的交付包。
9. `operations` 放上线后的反馈、变更和问题。

## 5. 必须补齐的治理规则

### 5.1 资料入库规则

资料分为四类：

| 类型 | 是否入库 | 说明 |
|---|---|---|
| 正式版 | 入库 | 有明确名称、版本、来源、适用范围 |
| 参考版 | 可入候选 | 可辅助问答，但答案需要弱化口径 |
| 草稿版 | 暂不入库 | 可保留在 raw，但不进入 selected |
| 废止版 | 不入库 | 仅保留存档，不参与问答 |

入库前必须确认：

1. 文档名称是否完整。
2. 是否有版本号或发布时间。
3. 是否能确认资料来源。
4. 是否能确认适用范围。
5. 是否存在同名、旧版、重复版。
6. 是否包含敏感内容或不应开放内容。

### 5.2 文档主数据字段

每份文档至少需要这些元数据：

| 字段 | 说明 |
|---|---|
| doc_id | 文档唯一编号 |
| title | 文档标题 |
| original_file | 原始文件路径 |
| doc_type | 制度、SOP、岗位职责、应急、安全、APP 操作等 |
| knowledge_domain | 知识域 |
| version | 版本号 |
| status | 正式版、参考版、草稿版、废止版 |
| source_owner | 资料来源部门 |
| business_owner | 业务责任人 |
| effective_date | 生效日期 |
| expire_date | 失效日期 |
| applicable_scope | 适用范围 |
| confidentiality | 权限级别 |
| parser_strategy | 解析策略 |
| selected_parser | 最终采用解析器 |
| update_cycle | 更新周期 |

### 5.3 知识域分类

第一阶段建议先按这些知识域管理：

| 知识域 | 覆盖资料 |
|---|---|
| 标准规范 | 服务区运营标准、企业标准、行业标准 |
| SOP 流程 | 岗位日常操作、场景处理流程 |
| 岗位职责 | 服务区经理、管理员、保洁、保安、加油站等 |
| 安全应急 | 安全生产、危化品、防汛、消防、事故上报 |
| 设施设备 | 水电、污水、二次供水、配电房、设备运维 |
| 经营服务 | 餐饮、超市、客房、投诉、满意度 |
| 信息系统 | APP 操作、系统使用说明 |
| 台账表单 | 记录表、检查表、字段说明 |

## 6. 文档解析策略

旧方案提出所有文档都跑 MinerU、Marker、MarkItDown。新版收敛为“主线解析 + 抽样对比 + 兜底补充”。

解析器定位：

1. `MinerU`：前期资料预处理主力，优先处理 PDF、扫描件、复杂版式资料和高价值标准资料。
2. `Marker`：抽样对比和兜底解析器，用于验证 MinerU 质量，或在 MinerU 失败、表格还原不佳时补充。
3. `MarkItDown`：DOCX 补充解析器，用于普通 Word 文档和 MinerU/Marker 不适合处理的 Office 文档。

推荐策略：

| 文档类型 | 首选解析 | 备选解析 | 说明 |
|---|---|---|---|
| PDF 标准/扫描件 | MinerU | Marker、pypdf | 关注页码、OCR、表格 |
| 普通 DOCX | MarkItDown | python-docx、Marker | 关注标题层级和段落完整 |
| 表格类 XLSX | openpyxl | Marker、MarkItDown | 优先结构化读取 |
| PPTX | MarkItDown | Marker | 关注标题和页面文本 |
| TXT/CSV/MD | 直接读取 | MarkItDown | 保留原始结构 |

抽样对比规则：

1. 每个知识域至少抽 2 到 3 份文档做 `MinerU + Marker` 对比。
2. PDF 类资料优先看 MinerU 输出，只有质量异常时才切换到 Marker。
3. DOCX 类资料优先看 MarkItDown 输出，必要时用 python-docx 做纯文本兜底。
4. 对高价值资料，如制度、SOP、安全应急资料，必须至少人工抽查一次。
5. 不要求首批所有文档三解析，避免处理成本过高。

解析结果必须保留：

1. 原文路径。
2. 解析器名称和版本。
3. 解析时间。
4. 内容字符数。
5. 标题数。
6. 表格数或表格行数。
7. 页码线索。
8. 异常和失败原因。

## 7. 质量评估标准

解析结果不能只看是否成功生成 Markdown，要按问答可用性打分。

核心评分维度：

| 维度 | 说明 | 权重建议 |
|---|---|---|
| 内容完整度 | 正文是否完整，是否缺页缺段 | 25% |
| 结构清晰度 | 标题、章节、条款是否保留 | 20% |
| 来源可追溯 | 页码、章节、原文路径是否保留 | 20% |
| 表格可读性 | 表格是否保留字段和值 | 15% |
| 噪声控制 | 页眉页脚、乱码、重复文本是否较少 | 10% |
| 问答适配度 | 是否适合切片和检索 | 10% |

入 `selected` 的最低要求：

1. 内容完整度不能低于 80 分。
2. 来源可追溯不能低于 70 分。
3. 正式资料必须人工确认。
4. 解析失败但资料重要的文档进入人工处理清单。

## 8. selected 定版规则

`selected` 不是“解析成功目录”，而是“可入库主版本目录”。

进入 `selected` 的条件：

1. 文档状态为正式版或确认可用参考版。
2. 元数据已补齐关键字段。
3. 解析质量通过最低门槛。
4. 至少完成抽样复核。
5. 不存在未处理的版本冲突。

`selected.md` 标准格式：

```md
# 文档名称

## 元数据

- 文档编号：
- 文档类型：
- 知识域：
- 版本：
- 状态：
- 来源部门：
- 适用范围：
- 生效日期：
- 最终解析器：
- 原始文件：

## 正文

<!-- page: 1 -->

正文内容
```

## 9. Chunk 生成与入库策略

chunk 生成是第一阶段的核心产物。切片不按固定字数硬切，优先按文档结构切。

| 文档类型 | 切片策略 |
|---|---|
| 制度规范 | 章、节、条款 |
| SOP 流程 | 场景、岗位、步骤、注意事项 |
| 岗位职责 | 岗位、职责项、任职要求、协同关系 |
| 应急安全 | 事件类型、预警等级、处置步骤、责任人 |
| APP 操作 | 功能模块、操作步骤、异常处理 |
| 表单台账 | 表名、字段、填写规则、使用场景 |

Chunk 必须包含：

| 字段 | 说明 |
|---|---|
| chunk_id | 切片唯一编号 |
| doc_id | 来源文档编号 |
| title | 来源文档标题 |
| section_path | 章节路径 |
| text | 切片正文 |
| source_ref | 来源章节或页码 |
| doc_type | 文档类型 |
| knowledge_domain | 知识域 |
| version | 文档版本 |
| permissions | 权限标签 |
| keywords | 关键词 |

生成后需要输出：

1. `chunks.jsonl`：用于向量化入库。
2. `chunks_preview.xlsx`：用于人工抽查。
3. `chunk_stats.json`：用于统计每份文档的 chunk 数量、平均长度、异常 chunk。

## 10. 向量库与 RAG 检索设计

第一阶段向量库只围绕 `AI+智能问答智能体` 服务，不先做多业务域大平台。

向量库目标：

1. 支持按问题召回相关 chunk。
2. 支持返回文档来源和章节线索。
3. 支持按知识域、文档类型、版本状态过滤。
4. 支持 RAG 检索效果评估。

建议第一版入库字段：

| 字段 | 说明 |
|---|---|
| chunk_id | chunk 唯一编号 |
| doc_id | 文档编号 |
| text | chunk 正文 |
| embedding | 向量 |
| title | 文档标题 |
| doc_type | 文档类型 |
| knowledge_domain | 知识域 |
| version | 版本 |
| source_ref | 页码、章节或条款 |
| source_file | 原始文件 |
| selected_file | selected.md 路径 |
| permissions | 权限标签 |

RAG 检索流程：

```text
用户问题
  ↓
问题改写 / 知识域识别
  ↓
向量检索 top_k
  ↓
元数据过滤
  ↓
召回结果重排
  ↓
构造回答上下文
  ↓
LLM 生成答案
  ↓
返回答案 + 来源引用
```

第一版 RAG 检索重点：

1. 能召回正确资料。
2. 能引用来源。
3. 能识别无资料问题。
4. 能减少无依据回答。
5. 能记录未命中问题，反哺资料和 chunk 优化。

## 11. QA 与评估设计

知识库上线前必须建立测试题库。

题库来源：

1. 武汉现场调研高频问题。
2. 标准化资料目录。
3. 服务区一线岗位常见问题。
4. 管理人员制度查询问题。
5. APP 操作问题。
6. 安全应急和危化品相关问题。

测试题至少分为四类：

| 类型 | 目标 |
|---|---|
| 精确问答 | 能从单一条款中找到答案 |
| 流程问答 | 能回答步骤、责任人和注意事项 |
| 对比问答 | 能区分不同岗位、不同场景要求 |
| 边界问答 | 对无资料、过期资料、权限外问题能拒答或提示 |

第一阶段验收指标建议：

| 指标 | 建议目标 |
|---|---|
| 高频问题可回答率 | 大于等于 80% |
| 答案来源引用率 | 大于等于 95% |
| 严重错误率 | 小于等于 5% |
| 无依据回答率 | 小于等于 3% |
| 权限外内容拦截 | 必须通过 |

## 12. AgentArts 接入边界

第一阶段 AgentArts 只需要调用知识库能力，不直接管理底层解析流程。

AgentArts 侧能力：

1. 接收用户问题。
2. 识别知识域和意图。
3. 调用知识库检索。
4. 组织答案。
5. 展示来源引用。
6. 记录用户反馈。

知识库侧能力：

1. 资料解析。
2. 元数据管理。
3. 切片和向量化。
4. 检索召回。
5. 来源追溯。
6. 知识包更新。

AgentArts 可接入形态：

| 形态 | 用途 |
|---|---|
| 知识包导入 | 快速接入第一版知识库 |
| API 检索服务 | 后续做统一知识库服务 |
| 工具调用 | 按知识域、权限、问题类型调用 |
| 反馈回流 | 收集未命中和低质量回答 |

## 13. 实施阶段拆分

### 阶段 0：知识库重建准备

目标：统一新计划，重建目录和规范。

产出：

1. 新版实施计划。
2. 目录结构。
3. 入库规范。
4. 元数据模板。
5. 质量评估模板。

### 阶段 1：资料盘点与定版

目标：弄清楚 raw 里到底有哪些资料，哪些能入库。

产出：

1. `documents_inventory.xlsx`
2. `duplicate_check.xlsx`
3. `version_check.xlsx`
4. 首批入库候选清单
5. 不入库和待确认清单

### 阶段 2：首批样本文档预处理

目标：先跑通 20 到 30 份高价值资料。

产出：

1. 解析结果。
2. 质量评估报告。
3. selected 样本。
4. chunk 样本。
5. chunks_preview.xlsx。

### 阶段 3：Chunk 与向量库构建

目标：把 selected 文档转为 chunk，并完成第一版向量化入库。

产出：

1. selected 主文档集。
2. chunks.jsonl。
3. chunk_stats.json。
4. 向量库集合。
5. seed QA。

### 阶段 4：RAG 检索测试与优化

目标：用题库验证召回和回答效果，修复资料、切片、召回和回答模板问题。

产出：

1. regression_questions.xlsx。
2. test_results.xlsx。
3. 问题修复清单。
4. RAG 检索报告。
5. 发布候选知识包。

### 阶段 5：上线与运维

目标：上线试用，并建立持续更新机制。

产出：

1. 上线说明。
2. 运维流程。
3. 反馈台账。
4. 版本更新记录。

## 15. 当前最先要做的 5 件事

1. 重建目录结构。
2. 生成 raw 资料盘点清单。
3. 定义元数据模板。
4. 从 raw 中选出 20 到 30 份首批样本文档。
5. 跑通“MinerU 预处理 -> selected -> chunk -> 向量库 -> RAG 检索”的最小链路。

## 16. 和旧方案的主要差异

| 旧方案重点 | 新方案调整 |
|---|---|
| 强调三解析器并行 | 改为 MinerU 主线、Marker 抽样对比兜底、MarkItDown 补充 DOCX |
| 强调 Kimi 二次处理 | 第一阶段先保证预处理、chunk、向量库、RAG 检索 |
| 缺少资料定版机制 | 增加正式版、参考版、草稿版、废止版 |
| 缺少元数据规范 | 增加文档主数据和 chunk 字段 |
| 缺少验收题库 | 增加 QA、回归测试和指标 |
| AgentArts 口径偏宽 | 收敛为知识包或知识检索服务接入 |
| 运维闭环不明确 | 增加变更、反馈、版本更新台账 |
